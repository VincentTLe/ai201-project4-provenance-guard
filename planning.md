# Provenance Guard — Planning

> A pluggable backend that classifies whether submitted creative text was written
> by a human or generated by AI, scores its confidence, surfaces a plain-language
> transparency label, and lets creators appeal a classification.
>
> **Design stance:** perfect AI detection is unsolved. This system optimizes for
> *honest uncertainty* over false certainty — and treats a false positive
> (calling a human's work AI) as the more damaging error.

### Architecture narrative — the path of one submission

A creator's platform sends a piece of text to **`POST /submit`** with the raw
`text` and a `creator_id`.

1. **Intake & rate limit.** Flask-Limiter checks the caller's request budget. If
   they're over the limit the request is rejected with `429` before any work
   happens. Otherwise a unique `content_id` (UUID) is minted.
2. **Signal 1 — LLM judge (semantic).** The text goes to Groq
   (`llama-3.3-70b-versatile`) with a classifier prompt. It returns a probability
   that the text is AI-generated (`0.0`–`1.0`) plus a one-line rationale.
3. **Signal 2 — Stylometric heuristics (structural).** Pure-Python metrics
   (sentence-length variance, type-token ratio, punctuation density) are computed
   locally and mapped to an "AI-likeness" score (`0.0`–`1.0`).
4. **Confidence scoring.** A weighted combiner merges the two signal scores into
   one calibrated `ai_probability`. That number is bucketed into one of three
   attribution verdicts: `likely_human`, `uncertain`, `likely_ai`.
5. **Transparency label.** The verdict + confidence select one of three
   plain-language label variants shown to readers.
6. **Audit log.** A structured record (timestamp, ids, verdict, combined score,
   each signal's score, status) is written to persistent storage.
7. **Response.** The endpoint returns `content_id`, `attribution`, `confidence`,
   the individual signal scores, and the `label` text.

The **appeal flow** is separate. A creator who disputes a verdict calls
**`POST /appeal`** with the `content_id` and their `creator_reasoning`. The system
updates that content's status to `under_review`, writes an appeal record to the
audit log (alongside the original decision), and returns a confirmation. No
automated re-classification — a human reviewer picks it up from the queue.

### Detection signals

Two signals that capture **genuinely different properties** — one semantic, one
structural — so their combination is more informative than either alone.

#### Signal 1 — Groq LLM judge (semantic / holistic)
- **Measures:** whether the text *reads* as machine-generated — coherence that is
  too smooth, hedging boilerplate, even register, absence of lived specificity.
- **Why it differs human vs AI:** LLM output tends to be fluent, balanced, and
  "safe"; human writing carries idiosyncrasy, opinion, and unpredictable detail.
- **Output:** `ai_probability` in `[0,1]` + short rationale string.
- **Blind spot:** heavily edited AI (a human polished it) reads human; and formal,
  polished *human* writing (academic, legal) can read as AI. The LLM also has no
  ground truth — it's a sophisticated guesser and can be confidently wrong.

#### Signal 2 — Stylometric heuristics (structural / statistical)
- **Measures:** statistical regularity of the prose:
  - **Sentence-length variance** — AI trends toward uniform sentence length.
  - **Type-token ratio** — vocabulary diversity; AI can be repetitive or, inversely,
    unnaturally even.
  - **Punctuation density** — humans use dashes, ellipses, parentheses irregularly.
- **Why it differs human vs AI:** AI text is statistically *smoother and more
  uniform*; human writing is *burstier and more variable*.
- **Output:** a single `ai_probability` in `[0,1]` combining the metrics.
- **Blind spot:** short texts (< ~40 words) don't have enough data for stable
  statistics. Certain human styles are naturally uniform (a terse minimalist, a
  poem built on repetition) and get flagged as AI. It measures *form*, never
  *meaning*, so it can't tell paraphrase from origination.

Because one signal is semantic and the other is structural, they fail in
*different* situations — the combiner can lean on whichever is more reliable and,
when they disagree, that disagreement itself becomes a signal of uncertainty.

> **Extensibility:** the combiner is a weighted vote, so adding a 3rd signal later
> (the *ensemble detection* stretch) is a config change, not a rewrite.

### The false-positive scenario (traced through the system)

**Scenario:** A non-native English speaker writes a formal, grammatically even blog
post entirely by hand. Its uniform sentence structure makes the stylometric signal
score it high (`~0.8` AI-like), and the LLM, seeing "too-smooth" prose, also leans
AI (`~0.7`).

- **Confidence scoring:** both signals point the same way, so the combined score is
  high — but the system is deliberately tuned so that "high AI score" maps to
  *"likely AI"* only above a conservative threshold, and even then the label never
  makes an absolute accusation. Where signals are merely *leaning*, the verdict
  falls into **`uncertain`**, not `likely_ai`.
- **Label:** the reader sees a hedged label ("Our tools suggest this *may* be
  AI-assisted; this is not a certainty") rather than a verdict that brands the
  writer a cheat.
- **Appeal:** the creator hits `POST /appeal` explaining they wrote it themselves
  and are a non-native speaker. Status flips to `under_review`, the appeal +
  original decision are logged together, and a human reviewer sees the full
  context. The creator is never stuck with a silent, unchallengeable verdict.

**Design consequence:** because a false positive is worse than a false negative on
a writing platform, thresholds are asymmetric — it takes *more* evidence to say
"likely AI" than to say "likely human," and the AI-side label is phrased as
possibility, not fact.

### API surface (the contract)

| Method & path | Accepts | Returns |
|---|---|---|
| `POST /submit` | `{ "text": str, "creator_id": str }` | `{ content_id, attribution, confidence, signals:{llm, stylometric}, label }` |
| `POST /appeal` | `{ "content_id": str, "creator_reasoning": str }` | `{ content_id, status: "under_review", message }` |
| `GET /log` | *(none)* | `{ entries: [ …recent audit records… ] }` |

- **Rate limiting** applies to `POST /submit` (limits & reasoning defined in M2/README).
- **Storage:** SQLite — a `content_records` table (holds status, verdict, scores per
  `content_id`) and an `audit_log` table (append-only decision + appeal history).

## Architecture

Two flows: **submission** (classify text → label) and **appeal** (contest a verdict).

```
                          SUBMISSION FLOW
                          ===============

  client
    │  POST /submit  { text, creator_id }
    ▼
┌─────────────────┐  over limit → 429
│  Rate limiter   │─────────────────────────────► client
│ (Flask-Limiter) │
└────────┬────────┘
         │ raw text (+ new content_id)
         ▼
┌─────────────────────┐        ┌───────────────────────────┐
│ Signal 1: LLM judge │        │ Signal 2: Stylometry       │
│   (Groq, semantic)  │        │   (pure Python, structural)│
└──────────┬──────────┘        └─────────────┬─────────────┘
   llm_score [0..1]                stylometric_score [0..1]
           └───────────────┬────────────────┘
                           ▼
                 ┌───────────────────┐
                 │ Confidence scoring│  combined ai_probability [0..1]
                 │  (weighted vote)  │  → verdict bucket
                 └─────────┬─────────┘
                           │ verdict + confidence
                           ▼
                 ┌───────────────────┐
                 │ Transparency label│  one of 3 label variants (text)
                 └─────────┬─────────┘
                           │ full decision record
                           ▼
                 ┌───────────────────┐
                 │  Audit log (SQLite)│
                 └─────────┬─────────┘
                           │ response body
                           ▼
                        client
        { content_id, attribution, confidence, signals, label }


                            APPEAL FLOW
                            ===========

  client
    │  POST /appeal  { content_id, creator_reasoning }
    ▼
┌──────────────────────┐   update status = "under_review"
│ content_records      │◄──────────────────────────────────┐
│  (SQLite)            │                                    │
└──────────┬───────────┘                                    │
           │ original decision + appeal reasoning           │
           ▼                                                │
┌──────────────────────┐   append appeal record ────────────┘
│  Audit log (SQLite)  │
└──────────┬───────────┘
           │ confirmation
           ▼
        client
   { content_id, status: "under_review", message }
```

**Narrative:** In the submission flow, raw text passes the rate limiter, fans out to
two independent detection signals (semantic LLM + structural stylometry), whose
scores are merged into one calibrated confidence, bucketed into a verdict, turned
into a reader-facing label, and recorded in the audit log before the response
returns. In the appeal flow, a creator's `content_id` + reasoning flips that
content's status to `under_review` and appends an appeal record beside the original
decision in the audit log, so a human reviewer sees the full history.
