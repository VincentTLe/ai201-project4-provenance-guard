# Provenance Guard — Planning

> A pluggable backend that classifies whether submitted creative text was written
> by a human or generated by AI, scores its confidence, surfaces a plain-language
> transparency label, and lets creators appeal a classification.
>
> **Design stance:** perfect AI detection is unsolved. This system optimizes for
> *honest uncertainty* over false certainty — and treats a false positive
> (calling a human's work AI) as the more damaging error.

---

## Milestone 1 — Architecture & Design

*(Milestone 2 expands this file with the full spec: uncertainty math, label text,
appeals detail, and edge cases. This section is the architecture foundation.)*

### Locked decisions (M1)

- **Storage:** SQLite — a `content_records` table + an `audit_log` table.
- **Detection signals:** **3** (ensemble), combined by a documented weighted vote —
  Groq LLM judge (semantic), stylometric heuristics (structural), and a lexical
  "AI-tell" signal (repetition / clichéd-phrase markers). This satisfies the base
  requirement (≥2) *and* the *Ensemble detection* stretch.
- **Stretch features targeted (all 4):** ensemble detection, provenance certificate,
  analytics dashboard, and multi-modal support. Each will get a planning.md update
  before it's built and a README write-up when done.

### Architecture narrative — the path of one submission

A creator's platform sends a piece of text to **`POST /submit`** with the raw
`text` and a `creator_id`.

1. **Intake & rate limit.** Flask-Limiter checks the caller's request budget. If
   they're over the limit the request is rejected with `429` before any work
   happens. Otherwise a unique `content_id` (UUID) is minted.
2. **Signal 1 — LLM judge (semantic).** The text goes to Groq
   (`llama-3.3-70b-versatile`) with a classifier prompt. It returns a probability
   that the text is AI-generated (`0.0`–`1.0`) plus a one-line rationale.
3. **Signal 2 — Stylometric heuristics (structural).** Pure-Python metrics
   (sentence-length variance, type-token ratio, punctuation density) are computed
   locally and mapped to an "AI-likeness" score (`0.0`–`1.0`).
3b. **Signal 3 — Lexical AI-tells (lexical).** A local scan scores the density of
   LLM-favored filler/hedging phrases and n-gram repetition (`0.0`–`1.0`).
4. **Confidence scoring.** A weighted combiner merges the three signal scores into
   one calibrated `ai_probability`. That number is bucketed into one of three
   attribution verdicts: `likely_human`, `uncertain`, `likely_ai`.
5. **Transparency label.** The verdict + confidence select one of three
   plain-language label variants shown to readers.
6. **Audit log.** A structured record (timestamp, ids, verdict, combined score,
   each signal's score, status) is written to persistent storage.
7. **Response.** The endpoint returns `content_id`, `attribution`, `confidence`,
   the individual signal scores, and the `label` text.

The **appeal flow** is separate. A creator who disputes a verdict calls
**`POST /appeal`** with the `content_id` and their `creator_reasoning`. The system
updates that content's status to `under_review`, writes an appeal record to the
audit log (alongside the original decision), and returns a confirmation. No
automated re-classification — a human reviewer picks it up from the queue.

### Detection signals

Three signals that capture **genuinely different properties** — semantic,
structural, and lexical — so their combination is more informative than any alone.
(Two would satisfy the base requirement; the third makes it an ensemble.)

#### Signal 1 — Groq LLM judge (semantic / holistic)
- **Measures:** whether the text *reads* as machine-generated — coherence that is
  too smooth, hedging boilerplate, even register, absence of lived specificity.
- **Why it differs human vs AI:** LLM output tends to be fluent, balanced, and
  "safe"; human writing carries idiosyncrasy, opinion, and unpredictable detail.
- **Output:** `ai_probability` in `[0,1]` + short rationale string.
- **Blind spot:** heavily edited AI (a human polished it) reads human; and formal,
  polished *human* writing (academic, legal) can read as AI. The LLM also has no
  ground truth — it's a sophisticated guesser and can be confidently wrong.

#### Signal 2 — Stylometric heuristics (structural / statistical)
- **Measures:** statistical regularity of the prose:
  - **Sentence-length variance** — AI trends toward uniform sentence length.
  - **Type-token ratio** — vocabulary diversity; AI can be repetitive or, inversely,
    unnaturally even.
  - **Punctuation density** — humans use dashes, ellipses, parentheses irregularly.
- **Why it differs human vs AI:** AI text is statistically *smoother and more
  uniform*; human writing is *burstier and more variable*.
- **Output:** a single `ai_probability` in `[0,1]` combining the metrics.
- **Blind spot:** short texts (< ~40 words) don't have enough data for stable
  statistics. Certain human styles are naturally uniform (a terse minimalist, a
  poem built on repetition) and get flagged as AI. It measures *form*, never
  *meaning*, so it can't tell paraphrase from origination.

#### Signal 3 — Lexical "AI-tell" markers (lexical / phrase-level)
- **Measures:** density of phrases that correlate with LLM output — filler
  transitions ("it is important to note", "furthermore", "in conclusion"), hedging
  boilerplate, and n-gram repetition.
- **Why it differs human vs AI:** LLMs over-use a recognizable set of connective and
  hedging phrases; human writing reaches for them far less predictably.
- **Output:** an `ai_probability` in `[0,1]` from marker density.
- **Blind spot:** easily gamed (delete the phrases and the score drops) and biased
  against genuinely formal/academic human writing that legitimately uses them. It's
  the weakest signal on its own — hence the *lowest weight* in the vote.

Because the three signals are semantic, structural, and lexical, they fail in
*different* situations — the combiner leans on whichever is more reliable and, when
they disagree, that disagreement itself becomes a signal of uncertainty.

> **Weighting (draft, finalized in M2):** LLM judge highest, stylometry mid, lexical
> lowest — a weighted vote so weights are a config change, not a rewrite.

### The false-positive scenario (traced through the system)

**Scenario:** A non-native English speaker writes a formal, grammatically even blog
post entirely by hand. Its uniform sentence structure makes the stylometric signal
score it high (`~0.8` AI-like), and the LLM, seeing "too-smooth" prose, also leans
AI (`~0.7`).

- **Confidence scoring:** both signals point the same way, so the combined score is
  high — but the system is deliberately tuned so that "high AI score" maps to
  *"likely AI"* only above a conservative threshold, and even then the label never
  makes an absolute accusation. Where signals are merely *leaning*, the verdict
  falls into **`uncertain`**, not `likely_ai`.
- **Label:** the reader sees a hedged label ("Our tools suggest this *may* be
  AI-assisted; this is not a certainty") rather than a verdict that brands the
  writer a cheat.
- **Appeal:** the creator hits `POST /appeal` explaining they wrote it themselves
  and are a non-native speaker. Status flips to `under_review`, the appeal +
  original decision are logged together, and a human reviewer sees the full
  context. The creator is never stuck with a silent, unchallengeable verdict.

**Design consequence:** because a false positive is worse than a false negative on
a writing platform, thresholds are asymmetric — it takes *more* evidence to say
"likely AI" than to say "likely human," and the AI-side label is phrased as
possibility, not fact.

### API surface (the contract)

| Method & path | Accepts | Returns |
|---|---|---|
| `POST /submit` | `{ "text": str, "creator_id": str }` | `{ content_id, attribution, confidence, signals:{llm, stylometric}, label }` |
| `POST /appeal` | `{ "content_id": str, "creator_reasoning": str }` | `{ content_id, status: "under_review", message }` |
| `GET /log` | *(none)* | `{ entries: [ …recent audit records… ] }` |

- **Rate limiting** applies to `POST /submit` (limits & reasoning defined in M2/README).
- **Storage:** SQLite — a `content_records` table (holds status, verdict, scores per
  `content_id`) and an `audit_log` table (append-only decision + appeal history).

## Architecture

Two flows: **submission** (classify text → label) and **appeal** (contest a verdict).

```
                          SUBMISSION FLOW
                          ===============

  client
    │  POST /submit  { text, creator_id }
    ▼
┌─────────────────┐  over limit → 429
│  Rate limiter   │─────────────────────────────► client
│ (Flask-Limiter) │
└────────┬────────┘
         │ raw text (+ new content_id)
         ▼
┌────────────────────┐ ┌──────────────────────┐ ┌──────────────────────┐
│ Signal 1: LLM judge│ │ Signal 2: Stylometry │ │ Signal 3: Lexical    │
│  (Groq, semantic)  │ │ (pure Py, structural)│ │  AI-tells (lexical)  │
└─────────┬──────────┘ └──────────┬───────────┘ └──────────┬───────────┘
   llm_score [0..1]      stylometric_score [0..1]   lexical_score [0..1]
          └──────────────────────┬┴───────────────────────┘
                                 ▼
                 ┌───────────────────┐
                 │ Confidence scoring│  combined ai_probability [0..1]
                 │  (weighted vote)  │  → verdict bucket
                 └─────────┬─────────┘
                           │ verdict + confidence
                           ▼
                 ┌───────────────────┐
                 │ Transparency label│  one of 3 label variants (text)
                 └─────────┬─────────┘
                           │ full decision record
                           ▼
                 ┌───────────────────┐
                 │  Audit log (SQLite)│
                 └─────────┬─────────┘
                           │ response body
                           ▼
                        client
        { content_id, attribution, confidence, signals, label }


                            APPEAL FLOW
                            ===========

  client
    │  POST /appeal  { content_id, creator_reasoning }
    ▼
┌──────────────────────┐   update status = "under_review"
│ content_records      │◄──────────────────────────────────┐
│  (SQLite)            │                                    │
└──────────┬───────────┘                                    │
           │ original decision + appeal reasoning           │
           ▼                                                │
┌──────────────────────┐   append appeal record ────────────┘
│  Audit log (SQLite)  │
└──────────┬───────────┘
           │ confirmation
           ▼
        client
   { content_id, status: "under_review", message }
```

**Narrative:** In the submission flow, raw text passes the rate limiter, fans out to
three independent detection signals (semantic LLM + structural stylometry + lexical
AI-tells), whose scores are merged into one calibrated confidence, bucketed into a verdict, turned
into a reader-facing label, and recorded in the audit log before the response
returns. In the appeal flow, a creator's `content_id` + reasoning flips that
content's status to `under_review` and appends an appeal record beside the original
decision in the audit log, so a human reviewer sees the full history.

## Milestone 2 — Specification

The five required design questions, answered concretely enough to implement against.

### 1. Detection signals — outputs & combination

All three signals output a single **`ai_probability` in `[0.0, 1.0]`** (0 = reads
fully human, 1 = reads fully AI), so they share one scale and combine cleanly.

| Signal | Type | Output | Weight |
|---|---|---|---|
| LLM judge (Groq) | semantic | `ai_probability` + one-line rationale | **0.50** |
| Stylometry | structural | `ai_probability` from 3 metrics | **0.30** |
| Lexical AI-tells | lexical | `ai_probability` from phrase/n-gram density | **0.20** |

**Why these weights:** the LLM reasons about meaning and is the most robust single
signal, so it leads. Stylometry is solid but fooled by naturally-uniform human
prose, so it's secondary. Lexical markers are the easiest to game and the most
biased against formal human writing, so they get the least say.

**Signal internals:**
- **LLM judge** — one Groq chat completion, `temperature=0`, a classifier prompt
  that asks for `Probability: <0..1>` and `Reason: <one line>`. Parsed defensively;
  on any parse/API failure it returns `0.5` (abstain — neither accuse nor clear).
- **Stylometry** — compute three metrics, map each to a `[0,1]` "AI-likeness" with
  the cutoffs below (linear interpolation between the two bounds; clamp outside),
  then **average the three** into the signal's `ai_probability`. Cutoffs are
  *starting values, calibrated against the four test inputs in M4:*
  - *sentence-length std-dev* (in words): `sd ≤ 3 → 1.0` (very uniform = AI-like),
    `sd ≥ 12 → 0.0` (bursty = human).
  - *type-token ratio* (unique/total words, first 100 words): `ttr ≤ 0.45 → 1.0`
    (repetitive = AI-like), `ttr ≥ 0.75 → 0.0` (diverse = human).
  - *punctuation variety* (count of distinct types among `. , ; : — ( ) ! ? … "`):
    `≤ 1 type → 1.0` (flat = AI-like), `≥ 5 types → 0.0` (varied = human).
- **Lexical AI-tells** — count LLM-favored filler/hedging phrases ("it is important
  to note", "furthermore", "in conclusion", "delve", "tapestry", …) and repeated
  n-grams per 100 words; map density to `[0,1]`.

**Combination:** `ai_probability = 0.50·llm + 0.30·stylo + 0.20·lexical`.

### 2. Uncertainty representation

**What a score means to the system.** `ai_probability` is "how AI-like the evidence
is." We derive a reader-facing **`confidence`** with **one formula for every verdict**
— how far the evidence sits from maximal uncertainty (0.5), rescaled to `[0,1]`:

```
confidence = 2 · |ai_probability − 0.5|      # clamp to [0, 1]
```

- `ai_probability = 0.50` → `confidence 0.00` (genuinely can't tell → **uncertain**)
- `ai_probability = 0.95` → `confidence 0.90` (strongly AI)
- `ai_probability = 0.05` → `confidence 0.90` (strongly human)
- `ai_probability = 0.71` → `confidence 0.42` (just over the AI line → *honestly* modest)

One formula avoids the earlier ambiguity of "what's the confidence of an *uncertain*
verdict" — it's simply the distance from 0.5, in whichever direction. `0.5` never
produces a coin-flip accusation; it produces the lowest possible confidence.

**Confidence bands (drive the label wording, §3):**
`confidence ≥ 0.60 → "high"` · `0.30–0.60 → "moderate"` · `< 0.30 → "low"`.
Because the AI verdict starts at `ai_probability ≥ 0.70` (`confidence ≥ 0.40`), a
borderline AI verdict reads *"moderate"*, not *"high"* — the label never overstates.

**Asymmetric thresholds (false-positive-averse).** It takes *more* evidence to call
AI than to clear a human:

| `ai_probability` | Verdict |
|---|---|
| `≥ 0.70` | `likely_ai` |
| `0.35 – 0.70` | `uncertain` |
| `< 0.35` | `likely_human` |

The "AI" band starts at a deliberately high 0.70; the "human" band is generous
(< 0.35). The middle is intentionally wide — we would rather say "uncertain" than
mislabel a person.

**Disagreement guard.** If the three signals disagree strongly — spread
(`max − min`) `> 0.40` — the verdict is **capped at `uncertain`** regardless of the
average, and confidence is reduced. Honest uncertainty beats a confident average
that papers over a real split between signals.

**Length floor.** Stylometry needs enough text to be stable:
- **`< 40` words:** drop the stylometry signal and renormalize the remaining weights
  over LLM + lexical (`0.50 : 0.20` → `0.714 : 0.286`).
- **`< 15` words:** too little signal to judge — **force `uncertain`** outright.

**How we'll test it's meaningful (M4):** run the four calibration inputs (clear AI,
clear human, formal-human, lightly-edited-AI) and confirm they land in *different*
bands with sensible confidences — not all clustered near 0.5, and no binary flip at
exactly 0.5.

### 3. Transparency label — the three variants (verbatim)

Plain language, honest about uncertainty, and — for the AI case — phrased as
*possibility*, never accusation. The verdict picks the variant; the `{level}` token
is filled from the confidence band (§2: high / moderate / low). These exact strings
ship in the response and README; the "high-confidence" variants below show `{level}`
resolved to **high**.

**High-confidence AI** (`likely_ai`, `{level}` = high):
> 🤖 **Likely AI-generated.** Our detection tools indicate this text was most likely
> produced with generative AI (confidence: high). This is an automated estimate, not
> a definitive judgment. If you wrote this yourself, you can appeal — see below.

**High-confidence human** (`likely_human`, `{level}` = high):
> ✍️ **Likely human-written.** Our detection tools found no strong signs of AI
> generation in this text (confidence: high). No attribution tool is perfect, but
> nothing here suggests this wasn't written by a person.

**Uncertain** (`uncertain`):
> ❓ **Attribution uncertain.** Our tools couldn't confidently tell whether this text
> was written by a person or generated by AI. Please treat this as inconclusive — we
> deliberately avoid labeling a human's work as AI when the evidence is mixed.

*(Template form: the AI and human variants read `(confidence: {level})`, so a
borderline `likely_ai` at confidence 0.42 displays "(confidence: moderate)". The
`uncertain` variant carries no level — being inconclusive is its whole message.)*

### 4. Appeals workflow

- **Who:** the creator of the content. In this backend an appeal is authenticated by
  possession of the `content_id`; real deployment would tie it to the `creator_id`'s
  authenticated session (noted as out-of-scope here).
- **What they provide:** `content_id` + `creator_reasoning` (free text — why they
  believe the classification is wrong).
- **What the system does, in order:**
  1. Look up the `content_record`; 404 if the `content_id` is unknown.
  2. Set that record's `status` → **`under_review`**.
  3. Append an `audit_log` entry of type `appeal` containing the original verdict,
     original confidence + all three signal scores, the `creator_reasoning`, and a
     timestamp — so the appeal sits *beside* the decision it contests.
  4. Return `{ content_id, status: "under_review", message }`.
- **No automated re-classification** — a human reviewer owns the outcome.
- **What a reviewer sees in the queue:** all `content_records` with
  `status = under_review`, each showing the submitted text, the original verdict +
  confidence, the three individual signal scores, and the creator's reasoning — the
  full context needed to uphold or overturn by hand.

### 5. Anticipated edge cases

1. **Repetition-heavy / minimalist human poetry.** Low vocabulary diversity and
   uniform line length make *stylometry* and *lexical* signals read it as AI, even
   though it's deliberate human craft. Mitigation: the LLM signal often recognizes
   creative intent, the disagreement guard then forces `uncertain` rather than
   `likely_ai`, and the creator can appeal.
2. **Non-native / formal human writing.** Even, "textbook" prose (ESL writers,
   academic/legal style) trips stylometry *and* the lexical phrase list. This is the
   core false-positive risk; the asymmetric 0.70 AI threshold + appeal path exist
   precisely for it.
3. **Lightly-edited AI output.** A human polished AI text — signals soften and it
   should land mid-range `uncertain`, which is the honest answer (it's genuinely a
   hybrid), not a failure.
4. **Very short submissions (< ~40 words).** Not enough text for stable statistics;
   stylometry is noise. Mitigation: below the length floor, down-weight stylometry
   and bias toward `uncertain`.

## AI Tool Plan

How each implementation milestone will use an AI coding tool, and how output is verified.

### M3 — submission endpoint + first signal (Groq)
- **Spec sections provided:** *Detection signals* (§1, LLM part) + the *Architecture*
  diagram + the *API surface* table.
- **Ask it to generate:** the Flask app skeleton with a `POST /submit` route stub, a
  `GET /log` route, the SQLite schema (`content_records`, `audit_log`), and the
  Signal-1 `llm_signal(text) -> {ai_probability, rationale}` function.
- **Verify:** call `llm_signal()` directly on 2–3 sample texts and inspect the
  returned shape/values *before* wiring into the route; confirm `/submit` returns
  `content_id` + a structured (placeholder) body and writes one audit row.

### M4 — second & third signals + confidence scoring
- **Spec sections provided:** *Detection signals* (§1, all) + *Uncertainty
  representation* (§2) + the diagram.
- **Ask it to generate:** `stylometry_signal(text)`, `lexical_signal(text)`, and the
  `combine(llm, stylo, lexical, word_count) -> {ai_probability, verdict, confidence}`
  implementing the exact weights, asymmetric thresholds, `confidence = 2·|p−0.5|`
  formula, disagreement guard, and length floor from §2.
- **Verify:** confirm the generated thresholds/weights match §2 exactly (AI tools
  drift here); run the four calibration inputs and check they land in different bands
  with sensible confidences; print per-signal scores to find any misbehaving signal.

### M5 — production layer (label, appeal, rate limit, full log)
- **Spec sections provided:** *Transparency label* (§3, verbatim variants) + *Appeals
  workflow* (§4) + the diagram.
- **Ask it to generate:** `make_label(verdict, confidence) -> str` returning the exact
  §3 text, the `POST /appeal` endpoint per §4, and Flask-Limiter config on `/submit`.
- **Verify:** ask it to emit all three label variants and diff against §3; submit
  inputs that reach each band; run an appeal and confirm `status` flips to
  `under_review` and the appeal row appears in `GET /log`; fire 12 rapid requests and
  confirm `200×10` then `429`.

*Rate-limit values (draft, finalized + justified in README at M5):* **10/minute,
100/day per IP** — covers a writer re-submitting edits in a burst while blocking
scripted floods.

## Stretch features — plan *(updated before each is built)*

1. **Ensemble detection** — satisfied by the 3-signal weighted vote above; the
   weighting and disagreement rule are the documented voting approach.
2. **Provenance certificate** — a "verified human" credential a creator earns via an
   extra step; stored on the `content_record` and surfaced in the label/response.
3. **Analytics dashboard** — a `GET /analytics` (or small HTML) reading the audit log:
   verdict distribution, appeal rate, + one more metric (e.g. mean confidence).
4. **Multi-modal support** — extend the pipeline to a second content type (image
   description / structured metadata) alongside text.
