# Provenance Guard

A pluggable backend for creative-sharing platforms that classifies whether
submitted text was written by a human or generated by AI — scoring confidence,
surfacing a plain-language transparency label, and giving creators a path to
appeal.

**Design stance:** perfect AI detection is an unsolved problem. Provenance Guard is
built to communicate **honest uncertainty** rather than force a binary verdict, and
it treats a *false positive* — labeling a human's work as AI — as the more damaging
error on a writing platform. That asymmetry shows up throughout: in the thresholds,
in the disagreement rule, and in how the labels are worded.

Full design rationale and the architecture diagram live in [`planning.md`](./planning.md).

---

## Architecture overview — the path of one submission

```
POST /submit {text, creator_id}
   │
   ├─ rate limiter (429 if over budget)          ← Flask-Limiter
   ├─ mint content_id (uuid)
   │
   ├─ Signal 1: Groq LLM judge      → ai_probability (semantic)
   ├─ Signal 2: stylometry          → ai_probability (structural)
   ├─ Signal 3: lexical AI-tells    → ai_probability (lexical)
   │
   ├─ combine() → weighted vote → combined ai_probability
   │            → verdict (likely_ai / uncertain / likely_human)
   │            → confidence = 2·|p − 0.5|
   │
   ├─ make_label(verdict, confidence) → reader-facing label text
   ├─ persist content_record + structured audit_log entry (SQLite)
   │
   └─ response {content_id, attribution, confidence, ai_probability, signals, label}

POST /appeal {content_id, creator_reasoning}
   └─ status → "under_review" → log appeal beside original decision → confirmation
```

A submission passes the rate limiter, gets a `content_id`, fans out to three
independent detection signals, whose scores are merged into one calibrated
confidence and bucketed into a verdict. The verdict + confidence pick a transparency
label; everything is written to the audit log before the response returns. An appeal
flips the content's status to `under_review` and logs the creator's reasoning beside
the original decision for a human reviewer.

```bash
python -m venv .venv
source .venv/Scripts/activate      # Windows (Git Bash);  .venv\Scripts\activate on cmd
pip install -r requirements.txt
cp .env.example .env                # then add your GROQ_API_KEY

```

Then open **http://127.0.0.1:5000/** in a browser for the demo web UI — paste text,

## Detection signals

Three signals capturing **genuinely different properties** — semantic, structural,
and lexical — combined by a weighted vote. Two would satisfy the requirement; the
third makes it an ensemble.

| Signal | Type | What it measures | Weight |
|---|---|---|---|
| **1. Groq LLM judge** | semantic | Whether the text *reads* as machine-generated (over-smooth coherence, hedging boilerplate, absence of lived specificity) | 0.50 |
| **2. Stylometry** | structural | Statistical regularity: sentence-length std-dev, type-token ratio, punctuation variety | 0.30 |
| **3. Lexical AI-tells** | lexical | Density of LLM-favored filler/hedging phrases + repeated-bigram ratio | 0.20 |

**Why these, and what each misses:**

- **LLM judge** reasons about *meaning*, which no statistic captures — so it leads.
  *Misses:* heavily edited AI reads human; polished formal human writing can read as
  AI; and it has no ground truth, so it can be confidently wrong.
- **Stylometry** is a cheap, deterministic structural check independent of the LLM.
  *Misses:* short texts lack the data for stable statistics; naturally uniform human
  styles (minimalist prose, repetitive poetry) look AI; it measures form, never meaning.
- **Lexical AI-tells** catch the specific phrase habits LLMs over-use.
  *Misses:* trivially gamed (delete the phrases), biased against formal human writing,
  and **one-directional** — it only scores *up* on marker density, so its absence
  pulls toward "human." It's the weakest signal, hence the lowest weight.

Because the three fail in *different* situations, their combination is more
informative than any one — and when they disagree, that disagreement itself signals
uncertainty (see below).

## Confidence scoring

**Combination.** `ai_probability = 0.50·llm + 0.30·stylometry + 0.20·lexical`
(renormalized over the active signals). Verdict comes from **asymmetric thresholds** —
it takes more evidence to accuse than to clear:

| `ai_probability` | Verdict |
|---|---|
| `≥ 0.70` | `likely_ai` |
| `0.35 – 0.70` | `uncertain` |
| `< 0.35` | `likely_human` |

**Confidence** is one formula for every verdict — distance from maximal uncertainty:

```
confidence = 2 · |ai_probability − 0.5|
```

So `0.5` yields confidence `0.0` (a genuine "can't tell"), never a coin-flip
accusation. Two guardrails enforce the false-positive-averse stance:

- **Disagreement guard (asymmetric):** if the signals' spread `> 0.40`, a would-be
  `likely_ai` is downgraded to `uncertain`. Split evidence can *revoke an AI
  accusation but never manufacture one* — it never forces a human verdict to uncertain.
- **Length floor:** `< 40` words drops stylometry (unreliable) and renormalizes;
  `< 15` words forces `uncertain`.

**How I validated it's meaningful.** I ran four deliberately chosen inputs (clear AI,
clear human, formal human, lightly-edited AI) through all signals and the combiner,
printing each signal separately. Scores land in different bands, span the range, and
don't cluster at 0.5 — and there's no binary flip at exactly 0.5.

**Two example submissions with noticeably different confidence** (actual scores):

| Input | llm / stylo / lexical | combined `ai_probability` | verdict | **confidence** |
|---|---|---|---|---|
| Casual human review ("ok so i finally tried that ramen place…") | 0.21 / 0.45 / 0.00 | **0.24** | `likely_human` | **0.52** |
| Lightly-edited AI ("I've been thinking about remote work… genuine tradeoffs…") | 0.62 / — / 0.00 | **0.44** | `uncertain` | **0.11** |

The casual human piece is scored with meaningful confidence (0.52) that it's human;
the ambiguous edited-AI piece is honestly flagged low-confidence (0.11) and left
`uncertain` rather than forced into a bucket — exactly the behavior the design intends.

## Transparency label — the three variants

The verdict selects the variant; the AI and human variants fill in a confidence word
(**high / moderate / low**) from the score, so a borderline result reads
"(confidence: moderate)" rather than overstating. The `uncertain` variant carries no
level — being inconclusive is its whole message. Verbatim text (shown with
confidence = high):

| Verdict | Label text |
|---|---|
| **High-confidence AI** | 🤖 **Likely AI-generated.** Our detection tools indicate this text was most likely produced with generative AI (confidence: high). This is an automated estimate, not a definitive judgment. If you wrote this yourself, you can appeal — see below. |
| **High-confidence human** | ✍️ **Likely human-written.** Our detection tools found no strong signs of AI generation in this text (confidence: high). No attribution tool is perfect, but nothing here suggests this wasn't written by a person. |
| **Uncertain** | ❓ **Attribution uncertain.** Our tools couldn't confidently tell whether this text was written by a person or generated by AI. Please treat this as inconclusive — we deliberately avoid labeling a human's work as AI when the evidence is mixed. |

## Rate limiting

Applied to `POST /submit` via Flask-Limiter, keyed by client IP:

```
10 per minute ; 100 per day
```

**Reasoning.** On a writing platform a genuine creator submits their own work
occasionally — even in an active editing session, re-submitting a revised draft a
handful of times a minute is plenty, so **10/minute** comfortably covers real use
while stopping a script from hammering the (LLM-backed, rate-limited, and costed)
endpoint. **100/day** caps sustained abuse — a single creator posting 100 distinct
pieces in a day is already implausible, and anything above that looks like automated
flooding or scraping. The two limits work together: the per-minute limit blocks
bursts, the per-day limit blocks slow-drip abuse that stays under the minute cap.

**Verified** (12 rapid requests):

```
request 1..10 -> 200
request 11    -> 429
request 12    -> 429
```

## Audit log

Every decision and appeal is written to SQLite as a structured entry; `GET /log`
returns recent entries as JSON. Sample (a classification and an appeal):

```json
{
  "event_type": "classification",
  "content_id": "925e63a3-045e-4910-a0d2-057de0146e85",
  "creator_id": "u-ai",
  "timestamp": "2026-07-01T04:15:20.123456+00:00",
  "attribution": "likely_ai",
  "confidence": 0.4066,
  "ai_probability": 0.7033,
  "llm_score": 0.82,
  "stylometric_score": 0.4931,
  "lexical_score": 0.7270,
  "signal_spread": 0.3269,
  "llm_rationale": "Overly generic phrasing and boilerplate suggest AI generation.",
  "scoring_notes": [],
  "status": "classified"
}
{
  "event_type": "appeal",
  "content_id": "925e63a3-045e-4910-a0d2-057de0146e85",
  "creator_id": "u-ai",
  "timestamp": "2026-07-01T04:15:28.508260+00:00",
  "event": "appeal_filed",
  "creator_reasoning": "I wrote this myself. I am a non-native English speaker and my style is formal.",
  "original_attribution": "likely_ai",
  "original_confidence": 0.4066,
  "original_ai_probability": 0.7033,
  "llm_score": 0.82,
  "stylometric_score": 0.4931,
  "lexical_score": 0.7270,
  "status": "under_review"
}
```

The appeal is logged **beside** the decision it contests, with the full signal
breakdown, so a human reviewer opening the queue (content records where
`status = under_review`) sees everything needed to uphold or overturn by hand.

## Appeals workflow

A creator who disputes a classification calls `POST /appeal` with the `content_id`
(returned by `/submit`) and their `creator_reasoning`. The endpoint:

1. Validates input and returns `404` if the `content_id` is unknown.
2. Flips that content's status to **`under_review`**.
3. Appends an `appeal` entry to the audit log containing the creator's reasoning
   **plus** the original verdict, confidence, and all three signal scores.
4. Returns a confirmation `{ content_id, status: "under_review", message }`.

There is **no automated re-classification** — a human moderator owns the outcome. The
review queue is simply the content records with `status = under_review`, each carrying
the full original decision and the creator's argument side by side.

## Known limitations

**Short, clean AI text without hallmark phrases is under-detected.** The lexical
signal is *one-directional*: it only scores upward on the density of LLM cliché
phrases and has no positive evidence for human authorship, so its `0.0` default
(weight 0.20) systematically pulls the combined score toward "human." A terse,
technical AI passage that avoids filler phrases ("furthermore", "it is important to
note") therefore tends to land `uncertain` rather than `likely_ai` — especially under
40 words, where stylometry is also dropped and the LLM judge is left doing most of
the work. This is a direct consequence of the signal's design, not a tuning issue.
It's also a *deliberate* trade: on a writing platform we would rather miss some AI
than falsely brand a human, so the bias points the "safe" way — but it is a real gap,
and a determined actor could exploit it by stripping cliché phrases from AI output.

A related case: **repetition-heavy human poetry** trips stylometry (low vocabulary
diversity, uniform line length) and can push the LLM toward an AI read; here the
disagreement guard and the appeal path are the safety net, but the margin can be thin.

## Spec reflection

**One way the spec helped.** Writing the three label variants and the asymmetric
thresholds *verbatim* in `planning.md` (§2/§3) before any code meant `make_label()`
and `combine()` were near-mechanical translations — there was no moment of "what
should a 0.62 actually say to a user?" during implementation, because that decision
was already made and written down.

**One way the implementation diverged.** The spec originally described the
disagreement guard symmetrically — "cap the verdict at uncertain regardless of the
average." During Milestone 4 calibration, a *clearly human* input got forced to
`uncertain` by that rule (one middling stylometry score created enough spread). That
contradicted the whole false-positive-averse thesis, so I changed the guard to be
**asymmetric** — it can revoke a `likely_ai` verdict but never downgrade
`likely_human` — and updated `planning.md` §2 to match. The divergence made the
system *more* faithful to its own stated principle, not less.

## AI usage

**1. First detection signal (Groq LLM judge).** I directed an AI tool to generate the
Flask skeleton and the `llm_signal()` function from `planning.md` §1. The draft asked
the model for a free-form verdict; I revised the prompt to force a bounded
`Probability: <0..1>` + `Reason:` format, added defensive parsing, and added a
**fail-safe abstention at 0.5** on any API/parse error — so a Groq outage produces
"uncertain," never a confident misclassification. The design intent was mine; the AI
produced the scaffold I then hardened.

**2. Confidence combiner.** I asked an AI tool to implement `combine()` from the spec.
Its first version applied the disagreement guard *symmetrically*. Calibration testing
showed this mislabeled a clearly-human input, so I **overrode** it to the asymmetric
guard described above — the AI's "reasonable-looking" logic silently diverged from the
project's core principle, exactly the kind of drift the spec warned about.

**3. Independent code review.** I ran an AI code-reviewer agent over the Milestone 4
code. It surfaced three real issues — a SQLite connection leak (`with conn` manages
the transaction but doesn't close the handle), a 500 crash on non-string JSON input,
and a punctuation-set deviation from the spec. I verified each against the actual code
and fixed all three; I did not accept the findings blindly (the agent also noted the
missing `/appeal` and rate limiting, which I confirmed were simply not-yet-built M5
scope, not bugs).

## Stretch features

- **Ensemble detection** ✅ — the 3-signal weighted vote above, with the documented
  weighting and the asymmetric disagreement rule as the voting/override logic.
- **Analytics dashboard** ✅ — `GET /analytics` reads straight from SQLite and returns
  detection patterns and health metrics: verdict distribution, appeal rate (+ raw
  appeal count and how many records are `under_review`), and — the extra metric — mean
  confidence and mean combined `ai_probability`, which together show at a glance
  whether the system is landing decisive or hedged verdicts. Example:

  ```json
  {
    "total_classifications": 3,
    "verdict_distribution": {"likely_ai": 1, "uncertain": 1, "likely_human": 1},
    "appeals": 1,
    "under_review": 1,
    "appeal_rate": 0.3333,
    "mean_confidence": 0.3723,
    "mean_ai_probability": 0.525
  }
  ```

- **Provenance certificate** ✅ — a "verified human" credential a creator earns through
  an extra verification step:
  - `POST /verify/start {creator_id}` issues a random **pledge sentence** + a
    `challenge_id`.
  - The creator must type that sentence back to `POST /verify/complete
    {challenge_id, response}`. Typing a specific phrase back is a deliberate human
    action — a lightweight, self-contained stand-in for a real identity/liveness check
    (a production system would swap in proper KYC). On a match, a certificate
    (`cert_id`, `creator_id`, `issued_at`, status `verified_human`) is stored.
  - **How it's displayed:** every `/submit` response now carries
    `provenance: {verified_human, certificate_id}`; the audit log records it; and the
    web UI shows a **✔ Verified Human Creator** chip beside the verdict. The credential
    is deliberately *independent* of the AI verdict — it certifies "this account proved
    it's a human creator," not "this specific text is human."

  Verified end-to-end: unverified submit → `verified_human: false`; wrong pledge → 400;
  correct pledge → certificate issued; subsequent submit → `verified_human: true`.

- **Multi-modal support** ✅ — `POST /submit` accepts an optional `content_type`
  (default `"text"`; also `"image_description"` for a caption / alt-text). It's a
  genuinely different modality, not just text again:
  - **Stylometry is dropped** (captions are short and structurally unlike prose).
  - **The LLM judge uses a caption-specific prompt** that looks for *caption* AI tells
    (exhaustive objective scene enumeration, generic aesthetic adjectives, "this image
    shows…" framing, no personal reaction) rather than essay tells.
  - **The vote is LLM-dominant (0.80 / 0.20 with lexical)** and the prose-mode
    disagreement guard and length floors are skipped, because the prose-oriented
    lexical signal is near-zero for captions and would otherwise create false
    disagreement or force short captions to "uncertain."
  - `content_type` is recorded in the response and audit log; everything downstream
    (thresholds, confidence, label, appeal) is unchanged.

  Verified: an AI-style caption ("A stunning image showcasing a vibrant sunset over a
  serene ocean…") → `likely_ai`; a human caption ("finally caught the sunrise after
  like 3 tries this week, freezing but worth it") → `likely_human`; an invalid
  `content_type` → `400`.
